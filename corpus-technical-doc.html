02 · Data layout

Everything lives in .corpus/

One hidden directory at the repo root. Committed to git (optional but recommended — teammates and CI get the brain for free).

.corpus/
├── graph.json          # nodes, edges, hierarchy — the map
├── state.json          # last update: commit hash, timestamp, file hashes
├── docs/
│   ├── _root.md        # project-level overview (rollup of top dirs)
│   ├── api/
│   │   ├── _dir.md     # rollup doc for api/
│   │   ├── auth.py.md  # file doc — mirrors repo structure
│   │   └── routes.py.md
│   └── frontend/…
├── changelog/
│   └── n_04f2.jsonl    # per-node history, append-only (post-core)
└── corpus.yml          # config: ignore rules, collapse pins, provider
  • docs/ mirrors the repo tree exactly — finding a file's doc requires no lookup table.
  • state.json stores a content hash per tracked file at last update — this is what staleness and incrementality diff against.
  • Ignore rules: .gitignore is respected automatically, plus a .corpusignore-style list in corpus.yml (default: lockfiles, migrations, build output, vendored code, binary assets).
03 · Graph schema & node identity

IDs that survive renames.

The problem: if node IDs are file paths, moving scraper.py to ingest/scraper.py destroys its history — the system sees a deletion and a creation. Identity must be decoupled from location.

// graph.json (abridged)
{
  "version": 1,
  "nodes": [
    {
      "id": "n_04f2",                    // stable random ID — never changes
      "path": "api/auth.py",             // current location — mutable
      "kind": "file",                    // file | dir
      "parent": "n_00a1",
      "lang": "python",
      "loc": 214,
      "symbols": ["hash_pw", "verify", "AuthMiddleware"],
      "importance": 4,                   // LLM rating from doc pass, 1–5
      "score": 0.81,                     // computed: fan-in + centrality + churn
      "stale": false,
      "doc": "docs/api/auth.py.md"
    },
    { "id": "n_00a1", "path": "api", "kind": "dir",
      "parent": "n_root", "collapsed": false, "doc": "docs/api/_dir.md" }
  ],
  "edges": [
    { "from": "n_04f2", "to": "n_07c9", "type": "imports" }
  ]
}

Rename survival

Git detects the rename

git diff -M --name-status between last-update commit and HEAD reports R097 scraper.py → ingest/scraper.py (97% content similarity).

Engine updates the path, keeps the ID

n_04f2's path field changes; ID, doc history, changelog, importance all persist. The doc file moves to mirror the new path.

Fallback for git-invisible moves

If a file was moved outside git's similarity threshold, content-hash matching in state.json catches identical files at new paths. A genuinely rewritten-and-moved file is, honestly, a new node — correct behavior.

Decision · vetoable

Node IDs are short random strings assigned at first sight, never derived from paths. Edges reference IDs, not paths. The frontend and MCP tools accept paths for convenience and resolve them to IDs internally.

Module rules (locked in conversation, restated)

  • Files are the atoms: docs, staleness, history, diff-mapping — all file-level, everywhere, no exceptions.
  • Directories are the view: rollup docs generated from child docs, shown collapsed or expanded.
  • Collapse heuristic is deterministic: collapse if file_count > 10 and median_loc < 80 (tunable in corpus.yml, per-directory pins override).
  • Display rank inside an expanded dir = computed score (fan-in, hub centrality, size, churn) adjusted by the LLM's 1–5 importance rating. Low scorers sit behind "show all". Wrong ranking is harmless by design.
  • The LLM never defines structure. Structure is computed; meaning is generated.
04 · The update pipeline

What corpus update actually does.

Snapshot diff

Compare working tree against state.json: git diff (with -M rename detection) plus content hashes. Output: sets of added / modified / renamed / deleted files.

Structural pass — deterministic, free

tree-sitter re-parses changed files → symbols, imports, calls. Graph nodes and edges updated. Deleted files' nodes removed; new files get fresh IDs.

Invalidation — compute the blast radius

Docs to regenerate = changed files ∪ their direct importers whose contract-facing symbols changed (see §05). Ancestor directory rollups of all affected files are queued for refresh.

Semantic pass — the only LLM step

For each invalidated file: prompt = file source + its old doc + the diff + one-hop neighbor summaries. Output = new doc (structured markdown) + importance rating + optional coupling notes. Calls batched, budget-capped.

Rollups & commit state

Affected _dir.md rollups regenerate from child docs (cheap LLM summarization). state.json updated with new hashes + commit ref. Staleness flags clear.

Decision · vetoable

Performance budget: an update after a typical session (≤15 changed files) must complete in under 60 seconds and fit inside the free tier's daily quota with room for 10+ updates/day. This budget is a design constraint, not an aspiration — it's why incrementality (M2) is core, not optimization.

M1 simplification

M1 skips steps 1 and 3 — full regeneration of everything, every run. Correct, slow, quota-hungry, fine. The pipeline is architected as above from day one so M2 slots into an existing shape rather than forcing a rewrite.

05 · Invalidation rule

When does a change over here stale a doc over there?

The subtle failure: change auth.py's function signature, and api.py's doc — which describes how it calls auth — is now wrong, with zero diff lines in api.py.

Corpus computes blast radius through the graph it already has:

  • Always re-doc: every file with actual diff lines.
  • Also re-doc: direct importers of a changed file, if the change touched exported/public symbols (tree-sitter tells us which symbols changed; the edge list tells us who imports them).
  • Never cascade further than one hop. Transitive invalidation (importers of importers) balloons cost for marginal accuracy. Second-hop drift is caught the next time those files change, and the staleness system means Corpus never claims false freshness anyway.
  • Directory rollups: refresh every ancestor of any re-doc'd file, root last.
Decision · vetoable

One-hop, symbol-gated invalidation. This is the accuracy/cost tradeoff at the heart of M2 — deeper cascades are a config flag later if drift proves annoying in practice, not a v1 default.

06 · Staleness mechanics

Amber, precisely defined.

  • A node is stale when its file's current content hash ≠ the hash recorded in state.json at last update. Computed against the working tree — uncommitted edits count, because mid-session is exactly when it matters.
  • A directory is stale if any descendant is stale.
  • The MCP server checks staleness at query time and attaches it to every response: "stale": true, "changed_since_doc": "+42/−7 lines". The agent knows to trust the doc less and read the file if precision matters. Corpus never silently serves a stale doc as fresh.
  • The frontend renders stale nodes amber; the file-watcher (a cheap mtime poll, not inotify complexity) flips them live as edits happen.
07 · MCP tool contract

Six tools. That's the whole bridge.

ToolInputReturns
corpus_overview()—Project rollup doc + top-level directory summaries + graph stats. The agent's first call every session.
corpus_doc(path)file or dir pathThe doc, plus staleness flag and symbol list. Dir paths return the rollup.
corpus_relations(path)file pathImporters, imports, and shared-symbol edges — the node's neighborhood.
corpus_find(query)keyword(s)Nodes matching by symbol name, path, or doc text (plain text search — grep-grade, deliberately not semantic).
corpus_changes(since?)optional ref/dateWhat's changed: stale files now; changelog entries once M6+ ships.
corpus_stale()—All currently stale nodes — lets the agent (or a hook) suggest running corpus update.

Every tool call also emits a fire-and-forget event to the event bridge: {tool, path→node_id, ts}. If no frontend is listening, events drop silently — the bridge is never load-bearing for the agent.

Decision · vetoable

Six tools, no more. Restraint is a feature: the developer's own internship finding (13 tools → 1 cut prompt tokens 76%) applies here — a fat tool list taxes every single agent prompt. New tools must displace an existing one or prove they can't be an argument on an existing one.

08 · Live event protocol

The wire behind the window.

The MCP server (stdio, talking to Claude Code) POSTs events to a tiny local sidecar (FastAPI, localhost:7077), which fans them out over websocket to any open frontend. Two processes because MCP's stdio lifecycle is owned by Claude Code — the sidecar outlives sessions and serves the static frontend build too.

// websocket frames, sidecar → frontend
{ "ev": "query",  "node": "n_04f2", "tool": "corpus_doc", "ts": 1721... }
{ "ev": "stale",  "node": "n_04f2", "on": true }
{ "ev": "graph",  "reason": "update_complete" }   // frontend refetches graph.json
  • query → node pulses teal for ~2s; collapsed ancestors glow so activity is visible at any zoom.
  • stale → node flips amber / back.
  • graph → soft reload of the map after corpus update finishes.
  • No buffering, no replay, no auth — localhost-only, ephemeral by design. If nobody's watching, nothing is stored.
09 · LLM layer

One wrapper, two providers, hard budgets.

  • Single llm.generate(prompt, max_tokens) wrapper; provider set in corpus.yml. Gemini 2.5 Flash primary; on 429, retry via Groq for small patches. Swapping providers is a config edit, never a refactor — free tiers change monthly and the architecture assumes it.
  • Doc prompts demand structured output: purpose (2–3 sentences), key symbols with one-liners, connections, gotchas, importance: 1–5. Parsed, validated, rejected-and-retried once on malformed output.
  • Every doc claim about a symbol must name the symbol it describes — cheap citation discipline that makes hallucinated claims spot-checkable and keeps the doc greppable.
  • Hard caps in config: max files per update, max tokens per call, max calls per day. Hitting a cap degrades gracefully: remaining files stay amber rather than the update failing.
  • Privacy rule: free-tier prompts may be used for model training. Personal repos: fine. Internship/company code: requires explicit clearance or a paid/local provider first. This is a config-level allowlist, not a memory-level promise.
10 · Milestones, now with acceptance criteria

Done means demonstrable.

M1
Static brain

corpus init on a real repo produces graph.json + full docs tree.✓ when: every non-ignored file has a doc; graph.json validates; run completes on a ~50-file repo within free-tier quota.

M2
Incremental updates

Diff → invalidation → patch. The hard one.✓ when: editing 3 files re-docs only those + gated importers + rollups; a rename preserves node ID and history; update ≤60s.

M3
Bridge + dogfood

FastMCP server, six tools, wired into Claude Code. Then a week of real use.✓ when: a fresh Claude Code session answers an architecture question via corpus tools without reading raw files; doc format revised from dogfood notes.

M4
Static window

React graph: collapse/expand, click-to-doc, importance-ranked reveal, amber staleness.✓ when: a 200-node repo renders smoothly and the graph is legible at default zoom.

M5
Live wire

Sidecar + websocket + pulse rendering.✓ when: asking Claude Code a question makes the consulted nodes visibly light up in real time. The demo.

Post-core, in order of cheapness: per-node changelog (diffs already flow through the pipeline), corpus explain for uncommitted diffs, health overlays, tour mode.

11 · Known risks

Where this can go wrong.

Doc hallucination

LLM docs served to the agent as truth can poison its work. Mitigations: symbol-citation discipline (§09), staleness honesty (§06), and dogfooding week explicitly hunting for confident-but-wrong claims.

Free-tier volatility

Quotas get cut without notice — Gemini's were slashed once already. Mitigation: provider wrapper + Groq lane + hard local budgets. Worst case: updates get slower, never broken.

M2 complexity spiral

Symbol-level change detection can rabbit-hole. Guardrail: if symbol-gating proves hard, fall back to "re-doc all direct importers" — costlier but correct — and ship.

Graph hairball at scale

Force layouts degrade past ~500 visible nodes. Mitigation: collapse-by-default heuristic + importance-gated reveal already cap visible nodes; edges render on hover/selection only above a threshold.

Scope creep

The project's owner has a documented additive-planning pattern. Guardrail: any new component idea goes to LATER.md; the six-tool cap and five milestones are the contract.
